# Research: 支付与资金流方案（多租户租售电商）

- **Query**: 货款分账（规避二清）、押金托管、租金计费的支付方案；给出 MVP 推荐 + 备选
- **Scope**: external（微信支付官方能力）+ mixed（结合 cool-admin/Midway/TypeORM 技术栈）
- **Date**: 2026-07-07

> 注：微信支付官方文档站（pay.weixin.qq.com/wiki、/docs）大量页面为前端 SPA 渲染，curl 直接抓取只能拿到 438KB 的 JS 壳（含 "JSAPI/分账/小程序" 关键词但正文需浏览器执行）。下文结论来自对微信支付 v3 API、电商收付通、分账、预付费/保证金产品规则的领域知识整理；接入前请以商户平台最新文档为准（资质与费率会变动）。

---

## 0. 三个核心难点拆解

| 难点 | 本质 | 关键诉求 |
|---|---|---|
| **货款分账** | 平台不能"代收属于商家的钱"再转付，否则构成**二清**（无证从事资金清算，央行/银监合规红线） | 钱要么直接进商家账户，要么由持牌通道（微信/银行）做合规分账 |
| **押金托管** | 租售结合特有：押金既不是平台收入也不是商家收入，是**履约担保**，需冻结、可扣（损坏/逾期）、可退 | 资金安全隔离 + 可控扣款，不能与货款混同 |
| **租金计费** | 按租期/周期计费与扣款（先付后租 / 周期续费 / 逾期自动扣） | 周期性扣款能力 + 续费/逾期状态机联动 |

---

## 1. 候选方案矩阵

### 方案 A：微信电商收付通（**合规首选 / MVP 推荐**）

电商收付通是微信支付为电商平台提供的**「二级商户 + 自动分账」**合规方案：平台作为**服务商**入驻，每个商家进件为微信的**二级商户（特店）**，消费者付款由微信**直接清分到各二级商户**，平台从分账规则抽佣金。

- **资金流**：消费者 → 微信 → 直接落账各商家二级商户户（平台佣金走分账接收方/手续费返还）
- **二清风险**：✅ 无。资金从不经过平台自有账户，由微信（持牌）完成清算
- **押金**：可走**「保证金」**子产品（电商收付通配套），二级商户冻结资金作为售后/纠纷担保；C 端押金也可用「预付费（预付充退）」
- **分账**：原生支持 `profit_sharing=true` + 分账接收方新增 + 分账下单，支持按比例/定额，最高分账比例可在进件时配置
- **资质要求**：平台需提供**电商类 ICP/EDI 或平台类资质**（营业执照、对公账户、电商平台资质证明），二级商户逐个进件（4-7 工作日，可走简化进件）
- **接入复杂度**：中高。需对接：①服务商进件 API（二级商户进件/KYC）②小程序 JSAPI 下单（`sp_mchid` + `sub_mchid`）③分账 ④退款 ⑤保证金 ⑥预付费。Node 侧用 axios + v3 SHA256-RSA 签名 + 平台证书/平台公钥验签
- **对模型影响**：
  - 商家（tenant）表需加 `wx_sub_mchid`（二级商户号）、进件状态、分账比例
  - 订单/支付单需存 `sp_mchid / sub_mchid / out_trade_no / transaction_id / profit_sharing` 标记
  - 资金账户表理论上**轻量**——平台不持资金，只存"应分账/已分账/可提现"对账数据（最终账在微信侧）
- **适配 cool-admin/Midway**：✅ 良好。把支付做成 **cool-admin 插件**（复用插件机制 + 多租户：插件实体继承 BaseEntity 取 tenantId），Service 封装签名/下单/回调；回调走 Midway 控制器 + TypeORM 事务更新订单状态。多租户天然契合——每个 tenant = 一个 sub_mchid

### 方案 B：微信支付分账（普通服务商，非电商收付通）

平台作为服务商，商家为特店进件，消费者付款先进**服务商主账户（待结算）**，再发起分账到商家。

- **资金流**：消费者 → 微信服务商主账户（待结算池）→ 分账 → 商家
- **二清风险**：⚠️ 灰区。钱先进服务商主账户，**理论上有二清嫌疑**；但因为是微信持牌分账产品、待结算资金不可挪用、且按规则分账，实务中**微信审核通过的「特约商户+分账」是被认可**的合规路径。比"平台代收+私转"安全得多
- **押金**：无原生保证金；押金仍需「预付费」或自建托管
- **分账**：`profit_sharing` 全套 API（与 A 相同）
- **资质**：服务商进件门槛低于电商收付通；特店进件同样需要
- **接入复杂度**：中。比 A 少了保证金/电商特有接口，分账 API 一致
- **对模型影响**：与 A 类似，但需多维护"待结算池 → 分账 → 提现"的中间状态对账
- **适配技术栈**：✅ 同 A
- **何时选它**：拿不到电商收付通资质（无 ICP/EDI 平台证），但又想合规分账 → 是 A 的降级版

### 方案 C：平台代收 + 手动/模拟提现（**MVP 极简 / 但二清**）

所有消费者的钱直接进**平台自有商户号**，平台定期（T+N）按订单把"商家应得"打给商家个人/对公账户。

- **资金流**：消费者 → 平台商户号 → 平台私转商家
- **二清风险**：🔴 **高**。这就是典型"二清"。规模一大、被举报或被监管抽查，风险显著（处罚/封号/法律责任）。仅适用于**早期极小规模 demo / 内测**，不可作为生产正式方案
- **押金**：平台自有账户里混同，需自己记台账（虚拟账户/科目隔离），不可挪用——但**没有银行级隔离**，仍属自管资金
- **租金计费**：可对接微信「委托扣款/周期扣款」（需签约）或纯手动
- **接入复杂度**：低。只对接一个商户号的 JSAPI 下单 + 退款
- **对模型影响**：最重——需自建完整的**虚拟资金账户体系**：商家账户余额、冻结/可用、提现申请/审核/打款、对账；押金要建虚拟冻结科目。等于平台自造一个"钱包清算系统"，复杂度被推到应用层
- **适配技术栈**：技术上最简单（一个微信商户号），但**合规债务最高**
- **何时选它**：仅作**沙箱/Demo/极早期**。一旦有真实交易量必须迁移到 A/B

### 方案 D：第三方聚合（Ping++ / 易宝 / 汇付天下 /连连）

聚合层 SDK 统一多渠道（微信/支付宝/银联），部分提供**合规分账账户体系**（如汇付天下「斗拱」、连连「综合支付+分账」、Ping++ 聚合）。

- **二清风险**：✅ 视通道而定。持牌机构的分账方案（汇付/连连/银盛等"账户分账"）合规；纯聚合（Ping++ 只做 SDK 聚合，不做资金清算）仍依赖底层通道
- **优点**：多渠道一套代码；部分提供**现成的虚拟账户 + 分账 + 提现**成体系方案，省自建
- **缺点**：多一层费用（交易费 + 提现费 + 分账费叠加）；接入要对接第三方私有 API；小程序场景**微信原生仍是首选**（聚合在小程序里最终也是调微信 JSAPI，意义打折）
- **接入复杂度**：中（多一层封装与对账）
- **对模型影响**：与 A 类似，但账户体系由第三方提供，平台只存对账副本
- **适配技术栈**：✅ 可行，但**与"小程序为主 + 单一微信支付"场景错配**——为单一微信渠道引入聚合层 ROI 低
- **何时选它**：①需要多渠道（支付宝/银联同样重要）②需要现成合规分账账户体系且不想自建 ③有合作渠道资源。**MVP 单微信小程序场景：不推荐引入**

### 押金专项方案（横切，适配 A/B/C）

| 方案 | 机制 | 可行性 | 说明 |
|---|---|---|---|
| **微信预付费（预付充退）** | 消费者先付一笔，存于微信侧，按需扣/退 | ✅ 推荐配合 A | 租赁押金理想载体，资金不进平台账户，扣退走 API |
| **微信支付分（先享后付/担保）** | 凭支付分额度担保，无需预付押金 | ⚠️ 看场景 | 体验好但需签约且对用户有分数门槛；适合「免押金租赁」营销卖点 |
| **电商收付通保证金** | 二级商户冻结自有资金做售后担保 | ⚠️ 方向不同 | 这是**商户**向平台交的保证金（约束商家），不是消费者押金，不要混淆 |
| **虚拟账户冻结（自建）** | 平台账上记"押金冻结"科目 | ⚠️ 仅 C 方案 | 无银行隔离，与二清同风险，不建议生产用 |

**押金推荐**：方案 A + 微信「预付费（预付充退）」做消费者押金；高信用用户叠加「微信支付分」做免押营销。

### 租金计费方案

- **先付后租**（一次性付清整租期）：直接复用 JSAPI 下单，无需特殊能力，**MVP 默认**
- **周期续费/自动扣**：微信「委托扣款 / 周期扣款（签约 + 扣款）」产品，需额外签约；用于月租自动续费。**MVP 可延后**，先用到期前提醒 + 手动续费
- **逾期**：状态机层面处理（订单转逾期 → 押金扣款 → 信用记录），资金层扣押金走预付费扣款 API

## 2. MVP 推荐方案

### 主推：**方案 A（微信电商收付通）+ 预付费押金 + 先付后租**

理由（按决策权重）：
1. **合规优先**：多商家 SaaS 天然命中"二清"风险场景，A 是唯一从根上消除二清的方案（钱不经平台）
2. **技术栈契合**：每个 tenant = 一个微信二级商户（sub_mchid），与 cool-admin 内置多租户**一一映射**，干净优雅
3. **押金有原生解**：预付费/保证金子产品覆盖租售结合的押金托管诉求，无需自建资金池
4. **分账原生**：平台佣金、商家货款、押金/货款分离都在微信侧完成，应用层只做对账

**MVP 落地切片**（建议拆解到 PRD/Spec）：
- [P0] 服务商进件 + 商家二级商户进件（后台表单 + 状态轮询）
- [P0] C 端小程序 JSAPI 下单（携带 `sp_mchid + sub_mchid`），支持**零售货款**与**租赁押金**两类订单
- [P0] 支付回调 + 订单状态机更新（租赁履约 / 零售履约分流）
- [P0] 退款（零售退货、租赁退押金）
- [P1] 分账下单（平台佣金抽成）+ 分账查询/对账
- [P1] 预付费押金：冻结（下单即冻）→ 扣款（损坏/逾期）→ 退还（正常归还）
- [P2] 委托扣款/周期扣款（月租自动续费）
- [P2] 微信支付分免押（营销增强）

### 备选 1：**方案 B（普通服务商分账）**
当拿不到电商收付通平台资质时降级使用。分账 API 与 A 一致，迁移成本低；缺保证金，押金需自建或用预付费。

### 备选 2：**方案 C（平台代收 + 模拟提现）— 仅 Demo/沙箱**
仅用于无资质阶段的演示与联调。**必须设置明确迁移里程碑**，进生产前转 A/B。押金在此方案下用"虚拟冻结科目"做账面隔离，标注"非银行隔离"风险。

### 明确不推荐
- **方案 D（聚合支付）** 在「单微信小程序」MVP 场景 ROI 低，多渠道需求出现后再评估
- **微信支付分做唯一押金方案**：受用户分数门槛限制，作为**营销增强**而非主押金路径

## 3. 对 cool-admin / Midway 技术栈的落点

### 插件化封装（复用 cool-admin 插件机制）
按 `plugin-architecture.md` 的结论，把支付做成 **cool-admin 插件** `plugin-wxpay-ecommerce`：
- 实体继承 `BaseEntity` → 多租户自动隔离（关键：商户/订单/分账记录都带 tenantId）
- Service 层：`WxpaySignService`（v3 签名/验签）、`WxpayOrderService`（JSAPI 下单）、`WxpayProfitService`（分账）、`WxpayDepositService`（预付费押金）、`WxpayWebhookService`（回调）
- 配置走 cool-admin 插件配置面板：`sp_mchid / apiV3Key / 商户证书序列号 / 平台证书 / 私钥`
- ⚠️ **C 端小程序热插限制**（PRD 已记）：C 端调用入口必须构建期纳入（uni 分包），插件热插仅限 admin/B 端配置

### 数据模型增量（TypeORM 实体）
```
MerchantTenantProfile  (tenantId 1:1)
  + wx_sub_mchid        二级商户号
  + wx_mch_status       进件状态 (PENDING/REJECTED/NORMAL/FROZEN)
  + profit_sharing_rate 平台分账比例

PayOrder  (订单聚合，零售/租赁共用)
  + out_trade_no        商户订单号
  + transaction_id      微信支付单号
  + sp_mchid / sub_mchid
  + business_type       GOODS(货款) | DEPOSIT(押金) | RENT(租金)
  + profit_sharing      是否待分账
  + status              状态机字段

DepositRecord  (押金托管，挂 PayOrder)
  + deposit_amount / frozen_at / released_at / deducted_amount
  + refund_trade_no

ProfitSharingBill  (分账账单)
  + receivers[]  [{type, account, amount, description}]
  + state        ACCEPTED/PROCESSING/FINISHED
```
资金账户表在方案 A 下**轻量**（最终账在微信侧，平台只存对账副本）；在方案 C 下则需重型虚拟账户体系（再次印证 C 的隐性复杂度）。

### 关键技术风险点
- ⚠️ **v3 签名/验签**：SHA256-RSA，需妥善管理商户私钥与微信平台证书（定期轮换、自动下载平台证书）。Midway 用 Node `crypto` + `node-forge` 或社区 `wechatpay-node-v3` SDK
- ⚠️ **回调幂等**：微信会重试回调，订单/押金/分账状态更新必须**幂等**（以 transaction_id + 状态判断为前置），TypeORM 事务包裹
- ⚠️ **多租户 + 回调**：回调进来时**没有用户 JWT/tenantId 上下文**，必须从 `sub_mchid` 反查 tenant → 手动 set tenant 上下文或绕过 Subscriber 写库（注意 PRD 警示的"原生 SQL 绕过租户过滤"风险，回调写入要走带 tenantId 的标准 Repository）
- ⚠️ **押金扣款时机**：损坏/逾期扣押金需有**证据留痕**（照片/工单），与订单状态机强联动，否则纠纷难举证
- ⚠️ **分账资金不足**：退款优先于分账，需在分账前校验可分账金额，否则分账失败需补款流程

## 4. 合规清单（落地前必做）
- [ ] 平台主体能否取得**电商类平台资质**（ICP/EDI），决定走 A 还是降级 B
- [ ] 微信**服务商**进件材料（营业执照、对公户、法人、客服电话）
- [ ] 商家**二级商户**进件流程在 B 端后台产品化（KYC、协议、状态轮询）
- [ ] **预付费/委托扣款**产品单独签约（非 JSAPI 默认开通）
- [ ] 资金对账日报（平台应分佣金 / 商家应到账 / 实际清分 三方对平）
- [ ] 明确**押金扣款规则**写入用户协议（损坏标准、逾期定义、扣款顺序），合规披露

---

## 5. 决策一句话

> **MVP 选「微信电商收付通 + 预付费押金 + 先付后租」**：每个 tenant 映射一个微信二级商户，钱由微信直接清分不进平台账户（消二清），押金走预付费托管，封装为 cool-admin 插件复用多租户隔离；拿不到平台资质时降级「普通服务商分账」；平台代收仅限 Demo 不可上生产；不引入聚合支付。

## Caveats / Not Found

- ⚠️ 微信官方文档为前端渲染 SPA，本研究的具体接口字段/费率/资质细则**未能在抓取中逐字核实**，落地实现时务必以商户平台最新文档为准（产品名、字段名、费率、签约门槛会变动）
- ⚠️ **费率**未列：服务商分账费率、电商收付通手续费、预付费/委托扣款的额外费率需在签约时确认，影响成本核算
- ⚠️ 「预付费/保证金/担保」在微信产品矩阵中名称与边界较细（保证金=商户交平台，预付费=消费者预付，支付分=信用担保），实现前需精确对齐产品名，避免选错
- 未深入第三方持牌分账（汇付斗拱/连连）的具体 API，若走 D 方案需单独补研究
