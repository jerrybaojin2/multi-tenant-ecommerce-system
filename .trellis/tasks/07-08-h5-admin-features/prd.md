# H5 与 Admin 功能点梳理

## 目标

系统性地梳理二期 H5 前端页面和管理后台（商家端 + 平台端）的功能点，为后续 PR1/PR3-PR9 实现提供清晰的功能范围和验收标准。

## 已知信息

### 来自主 PRD 的业务背景
- **业务模式**：多租户「租售结合」SaaS 平台
- **租户形态**：多商家入驻 SaaS，每个租户 = 独立商家
- **核心功能**：
  - 商品同时支持租赁和零售
  - 订单支持租买混单
  - 租赁流程：押金冻结 → 租期 → 归还/续租/逾期/买断
  - 零售流程：发货 → 自提/收货
- **数据隔离**：tenant_id + 应用层租户上下文 + RLS 兜底

### 前端技术栈（来自 spec）
| 端 | 技术栈 | 品牌 |
|---|---|---|
| C 端 MVP | uni-app Vue3 + Vite + TS + wot-design-uni + Pinia | 微信小程序 |
| H5 二期 | 待定 | 待定 |
| Admin | Next.js + TypeScript；业务流程全部调用自研 Midway.js 后端 API，不放入 Next.js API routes | merchant（商家） + platform（平台） |

### 当前 PR 序列
- PR0: ✅ 自研 Midway 基座 + 多租户隔离验证
- PR1: 🔜 三端 walking skeleton
- PR3-PR9: 业务功能逐步落地

### 关键约束
- 租户上下文不可相信客户端字段
- C 端请求必须注入 X-Tenant-Id
- Admin menus/routes/permissions 后端驱动
- H5 是二期规划，MVP 仅微信小程序

## 假设（待验证）

- H5 面向 C 端用户，作为小程序的补充渠道
- Admin 双品牌：商家管理自己的商品/订单/资金，平台管理商家入驻/套餐/分账
- H5 和小程序功能基本对等，但可能有差异化特性

## 技术映射（API + 数据模型）

> 详细映射见 [`research/api-data-model-mapping.md`](research/api-data-model-mapping.md)

### 数据模型覆盖模块
goods、inventory、order、aftersale、payment、funds、promotion、logistics、customer-service、content、consumer-user、rbac、platform、rental（预留）

### API 接口覆盖
- **C 端** `/app/consumer/**`：认证、用户/地址、商品、收藏/浏览、购物车、订单、支付、售后、优惠券/会员、物流、客服、内容
- **商家端** `/admin/merchant/**`：商品、订单、库存、售后、资金、营销、客服、物流、店铺、权限
- **平台端** `/admin/platform/**`：商家、套餐、结算、数据、内容、审计、投诉、物流、权限

### 关键设计原则
- 租户表统一 `tenant_id`；平台表不隔离（`merchants`/`packages`/`settlements`/`audit_logs`）
- 商品 `sales_type(sale/rent/both)` 与订单 `type(sale/rent/mixed)` 字段预留租赁扩展，MVP 仅 `sale`
- 快递鸟集成走 `integrations/` 适配器，物流轨迹/面单/状态推送不污染领域实体
- 菜单/权限后端驱动，前端只渲染（符合 spec/frontend 约定）
